```js
const nested = [
  [1, 2, 3],
  [4, 5, 6],
  [7, 8, 9],
]

const copied = nested
```

이게 생각처럼 copied와 nested에 각각 서로 다른 배열에 저장되지 않는다. 아래에서처럼 nested에 변형을 가하면 그 변경사항이 copied에도 그대로 적용된다.

```js
nested.pop()

console.log(copied)
// 기대한 copied의 값
// [
//   [1, 2, 3],
//   [4, 5, 6],
//   [7, 8, 9],
// ];
// 출력되는 copied의 값
// [
//   [1, 2, 3],
//   [4, 5, 6],
// ];
```

그 이유는 `const copied = nested;` 이 부분에서 얕은 복사가 됐기 때문이다. 이 부분을 이해하기 위해서는 자바스크립트가 어리석은 인간들을 위해 우리 몰래 해주고 있던 배려를 알아야 한다.

```js
const nested = [
  [1, 2, 3],
  [4, 5, 6],
  [7, 8, 9],
]
console.log(nested)
// 자바스크립트의 배려 - 우리에게 보이는 값
// [
//   [1, 2, 3],
//   [4, 5, 6],
//   [7, 8, 9],
// ];
// 사실 자바스크립트가 가지고 있는 값 (예시)
//
```

위에서 말한 얕은 복사란, 복사는 했지만 저 주소값이 복사가 이뤄졌다는 말이다. 결국 nested나 copied나 같은 주소가 써진 종이를 손에 들고 있는 것과 같다.

## JSON.parse(JSON.stringify(객체))

### 장점

간편하다.

```js
const copied = JSON.parse(JSON.stringify(nested))
```

객체 자체를 JSON문자열로 바꿨다(stringify)가 다시 객체로 해석(parse)하므로 기존에 있던 주소값들은 전혀 상관이 없게 되고, 완전히 새로운 주소들에 써진 객체가 된다. 어떻게보면 인간이 하는 사고방식과 가장 유사한 방법이다.

### 단점

1. 데이터의 종류에 따라서는 손실 또는 변형될 수 있다.
2. 이 가능성을 에러 메시지나 경고 메시지 등으로 전혀 알려주지 않는다.

```js
const nested = {
  name: undefined,
  sayHello: () => console.log("Hello!"),
}
const copied = JSON.parse(JSON.stringify(nested))

console.log(copied)

// 기대한 copied의 값
// {
//   name: undefined,
//   sayHello: () => console.log("Hello!")
// };

// 출력되는 copied의 값
// {}
```
